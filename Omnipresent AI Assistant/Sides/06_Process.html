Aldrich · Process
Section 2.1 — Engineering principles

Six rules every PR is held to.

  • Safety layer first. No action ships before its type is in the Sandbox Classifier, given an undo path, covered by adversarial tests.
  • Every PR ships with tests. Integration tests against sandbox accounts (Gmail, Slack, Notion) — not just unit tests.
  • LLM-dependent code has a non-LLM fallback. If the LLM is unavailable or wrong, degrade gracefully.
  • Latency budgets are enforced in CI. A PR that regresses P99 by >10% on any critical path fails CI.
  • Every action has a compensation. If it can't be undone, it has a compensating transaction (apology email, etc.).
  • No experimental ML in production write paths. Experimental models can inform — they can't decide.

Aldrich · Process
Section 2.4 — Non-negotiable gates

A feature cannot ship unless ALL of these are true.

  • Action type registered in the Sandbox Classifier (or marked read-only).
  • Has an undo path OR a compensating transaction OR is in the biometric-required tier.
  • ≥5 integration tests against real sandbox accounts.
  • P99 latency inside the budget (200ms local, 1500ms cloud).
  • Data access uses minimum-scope OAuth permission.
  • Failure mode has a user-visible explanation.
  • Adversarial test suite includes ≥3 cases attempting to abuse the new capability.
  • A kill switch exists — feature flag that disables in seconds.

Section 3

Runtime process.

What happens 5–15 times per day per active user.

08
Aldrich · Process
Section 3.1 — The full data journey

Five stages, voice → action.

  1. 01 User initiates — wake word OR push-to-talk OR scheduled trigger OR incoming event.
  2. 02 Local processing — Silero VAD on-device → Gemma 2B intent classifier (~180ms TTFT) → simple intent handled locally; complex sent encrypted to cloud.
  3. 03 Cloud reasoning — pgvector retrieves top-20 memories. ReAct loop: Reason → Act → Observe → repeat (max 8 steps, 30k token budget).
  4. 04 Safety + biometric — sandbox classifier checks category/risk/budget; high-risk requires Face ID/PIN.
  5. 05 Execution — action fires. Action Record stored with forward + inverse payloads. 30-second undo window. Committed if not undone.
09
Aldrich · Process
Section 3.2 — The ReAct loop in plain English

Reason → Act → Observe → repeat.

  • Reason. "What is the user actually asking for?"
  • Act. Pick a tool from the allowed registry (e.g., calendar.getFreeBusy) and call it.
  • Observe. What did the tool return? Is it what I expected?
  • Reason. "Given what I now know, what's the next step?"
  • Repeat 2–4, up to 8 times max.
  • Produce. Generate the final SDUI JSON output (drafted email, confirmation card, etc.).

Each step is an LLM call. Each tool call hits a real API. Whole loop runs inside a Go FSM that enforces the 8-step cap and 30k token budget.

Section 3.5 — Privacy boundary at runtime

What stays on-device. What leaves. What's never stored.

Stays on-device

  • Raw audio
  • Full screen content
  • Biometric data
  • Local-app content not explicitly sent

Leaves device (encrypted)

  • Intent summaries
  • Email metadata; full body only on approval
  • Calendar event titles + times
  • Approved action payloads

Never stored after session

  • Raw audio (deleted post-transcription)
  • Rejected drafts (unless user opts in)

Aldrich · Process
Section 4.1 — Onboarding (first 48 hours)

Calibration first. Action later. Trust built through visible intelligence.

  • Hour 0: Install → 5-question calibration interview (~3 min) → OAuth for 1–3 integrations.
  • Hours 1–48: Observation phase. Daily "I noticed…" briefings, no actions proposed. Trust is built through visible intelligence without risk.
  • Day 2: First proposed action — low-stakes, reversible, educational. ("Buffer 15 min before your 9am calls?")
  • Week 1: Suggestions only.
  • Week 2: Low-stakes actions with approval (drafts, schedule suggestions).
  • Week 3+: Routine auto-actions; exceptions escalated.
  • Month 2+: High-frequency routine auto-executed; destructive/financial always gated.

Aldrich · Process
Section 4.2 — The daily rhythm

Loud at 7am. Silent in deep work. A quiet summary in the evening.

  • 6:00–7:30am — Aldrich processed overnight: scored emails, calendar conflicts, Slack mentions, sleep data. Lock-screen briefing card: "3 things need your attention."
  • ~45 seconds of scrolling — Approve, Edit, or skip. Three actions handled before coffee.
  • Throughout the day — User in deep work. Aldrich silent. Real-time intervention only for: deep-work conflict, VIP message, time-sensitive commitment.
  • Evening summary — "11 actions today. 9 auto-approved. 2 needed your input."

Aldrich · Process
Section 4.3 — Approval flow

Every approval card carries: what, why, source, affordances.

Anatomy of an SDUI Action Card:

  • Title: "Reschedule call with Sarah to 3pm?"
  • Why (one line): "Current 2pm slot conflicts with Deep Work block."
  • Source attribution: "Sarah's email at 9:14am."
  • Affordances: [Edit] [Approve] [Reject].

Approve flow: tap Approve → Face ID (if high-risk tier) → action fires → 30-second undo banner → committed after 30s.

Edit flow: tap Edit → inline editor → tweak → resume Approve flow.

16
Aldrich · Process
Section 4.4 — Error recovery

Every card has a "Something went wrong" button.

  • User describes what happened — preset options + freeform: "Wrong recipient" / "Wrong time" / "Wrong decision" / "Other."
  • Aldrich offers compensation immediately: reversible → one-tap undo; not reversible → generated apology; serious → escalation ticket to human support.
  • System side: incident logged with user_reported = TRUE; if >5 incidents in 24h, T&S team alerted; feedback added to next-iteration training pool.
  • Within 24h: explanation email — "Here's what happened. I've updated my approach."

Aldrich · Process
Section 5.2 — Incident response

Acknowledge → investigate → contain → communicate → resolve → postmortem.

  • Acknowledge within 5 min for SEV-1.
  • Investigate — Datadog query, ReAct chain logs, single-user vs. system-wide.
  • Contain — system-wide: kill switch + feature flag off. Single-user: freeze actions, notify user.
  • Communicate — single-user: in-app within 1h. System-wide: status page + X within 15 min.
  • Resolve — root cause + fix deployed (bypassing normal cadence if SEV-1).
  • Postmortem — blameless, written within 5 business days. Adversarial test suite updated.
20
Aldrich · Process
Section 5.4 — Kill switch hierarchy

Three levels of emergency kill.

  • User-level. User hits kill switch in app → all their active ReAct loops halt within 5s (UDP broadcast). Pending approvals dismissed. OAuth tokens flagged "paused."
  • Feature-level. Product team disables a capability (e.g., auto-send email) via flag → all instances halt within 60s globally.
  • System-level. Engineering on-call hits the system kill → all new ReAct loops rejected at API gateway; in-flight loops complete but no new ones start. Absolute emergency only.

Section 5.3 — Weekly operational ritual

Three meetings, one rhythm.

  • Monday — Weekly metrics review. AAPUW (north star), Day-30 retention, error rates, LLM cost/user. Red/yellow/green scorecard.
  • Wednesday — Adversarial test review. Any failures in the 500-case corpus? Any new attack patterns from production?
  • Friday — "Ship it" review. What ships next week? Go/no-go on each feature flag.

Section 8

Safety — the immune system.

Three lines of defense. Quarterly red team. Continuous user-reportable safety.

23
Aldrich · Process
Section 8.1 — Three lines of defense

Defense in depth. No single layer is trusted alone.

Line 1 — Input

  • Deterministic sandbox classifier on every input
  • LLM output schema-validated against tool registry
  • External content tagged "untrusted," through Llama Guard

Line 2 — Action

  • Risk levels: low/medium/high/critical
  • High/critical → biometric gate
  • Action budgets per user per hour
  • Circuit breaker on anomalies

Line 3 — Reversibility

  • 30-second undo window for reversible actions
  • Compensating transactions for irreversible
  • Full audit log for forensics
  • Account-level pause
24
Aldrich · Process
Section 8.2 — Adversarial testing

Quarterly red team. 100+ scenarios. Every win becomes a test.

  • Internal red team (security engineer + outside contractor) attempts 100+ attacks: prompt injection via email, impersonation, privilege escalation, data exfiltration, bulk-action abuse.
  • Every successful attack becomes a test case in the adversarial suite (grows to 500+ cases).
  • Every weakness becomes a ticket with SLA — critical: 48h, high: 1 week, medium: 1 month.
  • Results reviewed by leadership and published internally.
User-reportable safety

Every action card has a "Report a concern" link. Reports route to T&S queue with 24h SLA. Patterns trigger capability review. Repeated user policy violations → suspension.

Aldrich · Process
Section 9 — Data lifecycle

Creation → active → archive → deletion → export.

  • Creation. User authorizes integration → tokens encrypted with user-specific key → stored in vault. Data flows from integrations → classified → summarized → memory hub.
  • Active use. Memory retrieved on every ReAct loop (~50–300ms). Frequently retrieved stays warm; rarely retrieved cools off.
  • Archival (after 90d inactivity). Cold memories → S3 + DynamoDB. Resurrectable on demand (~200ms extra).
  • Deletion (GDPR right to erasure). Cascade across 8 tables + S3 prefix + pgvector + Kafka offset reset. Completes within 30 days. Confirmation email. OAuth tokens revoked with all third parties.
  • Account export (GDPR portability). Background job → JSON → signed URL via email, valid 7 days.
