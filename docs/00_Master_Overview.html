Aldrich · Master Overview · Part 2
How it works — in plain language

Five steps that run 5–15 times per user per day.

No AI magic. Each step is a well-understood component, assembled in a specific order. Users don't see these steps — they see a briefing, a suggestion, or a button to approve.

1 — User says or taps something

Voice command, tap on a notification, or a scheduled morning trigger fires. App captures the request.

2 — A small AI on the phone decides: simple or complex?

Simple (e.g., "what's on my calendar?") → handled locally in under a second. Complex → sent encrypted to cloud.

3 — The cloud brain pulls relevant memory and reasons

Claude (our cloud AI) reads the top-20 memories related to the request, plans the action, and calls the right tools (Calendar, Slack, Gmail).

4 — Safety layer checks before anything happens

Is this action reversible? Does it need Face ID approval? Is the user over their action budget? Risky actions pause and ask.

5 — Action fires with a 30-second undo window

Email sent? You have 30 seconds to undo. Calendar changed? Same. Every action is logged and reversible.

The technical bet, de-jargoned

On-device AI (the small model on the phone) = instant, free, private. Cloud AI (Claude) = smart reasoning but costs money per request. Safety layer = a classifier that checks every outgoing action against a rule list. None of these are exotic — the work is stitching them together reliably.

Aldrich · Master Overview · Part 3
The bootstrap question — can we do this without raising?

No. Even at PK base, bootstrapping breaks by month 4.

The PK cost base helped us — a lot. It dropped Y1 burn from $2.5M to $470K. But that's still not something one or two founders can personally fund for a year, while pre-product revenue is $0.

What bootstrapping actually covers

  • ~$100–150K typical founder self-funding capacity
  • ≈ 3–4 months of burn at our rate
  • Enough to build the safety foundation + start the Memory Hub
  • Not enough to ship V1, run beta, or launch

What happens if we try

  • Month 4: cash out. Product is 30% done.
  • No marketing budget → no launch signal → no signups
  • No contingency → first surprise ends the project
  • Founders take unsustainable debt or shelve the project
Message to the team

The PK cost structure is a real advantage — but it's an efficiency lever, not a funding replacement. The math is clear: we need external capital to do this properly. Telling ourselves "we can bootstrap" is the fastest way to build 40% of the product and discover we can't finish it.

What we should do instead

Raise $1M seed early. Ideally from our network + one lead angel. Use the PK cost advantage to deliver outsized progress per dollar raised — that's what makes the Series A possible at good terms 12 months later.

Challenge 6 of 6 — and why MCP changes it

Commoditization was the existential risk. MCP flips it.

Six months ago we'd have written this slide as pure downside. "Every tool ships its own AI layer, the user never feels enough cross-tool pain, we die." Then Anthropic shipped MCP (Model Context Protocol) — and it changed what's possible.

The old fear

  • Slack AI, Notion AI, Gmail AI each ship their own memory
  • Users never stitch them together
  • Cross-tool pain never gets acute enough for a paid layer
  • Aldrich becomes a great product nobody needs enough to pay for

What MCP changes

  • Our memory becomes an MCP server — a universal memory endpoint
  • Claude Desktop, Cursor, future ChatGPT (with MCP support) — all can read it
  • User brings Aldrich's memory to every AI tool they use
  • Our position becomes: the memory substrate that any AI tool can plug into
The reframe

Instead of competing against every tool's in-house AI, we make our memory available to all of them via MCP. The more AI tools the user uses, the more valuable Aldrich becomes. Commoditization of AI tools helps us — because users accumulate more places where our memory proves useful. That is a moat only we can build right now, because MCP is new and we're early.

Pick two to start. One that generates revenue fast, one that drives distribution. The remaining three wait until the first two prove out. Three simultaneously is a distraction, not a strategy.
